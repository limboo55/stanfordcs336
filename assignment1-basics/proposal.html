Building and Training a language model from Scratch

A Project Proposal Based on Stanford's CS336

Xu Fang

Relevance & Interest

The "API-Only" Confusion

When working on NLP assignments, I've often felt confused. I use high-level libraries to call powerful models, only interacting with the API.

I don't get to see how they work, why they are designed that way, or what happens inside. It feels like a 'black box'.

The Importance of Fundamentals

To truly understand language models, building them from the ground up is essential. This project isn't just using a Transformer; it's about implementing one.

This "from scratch" approach is the best way to gain deep, foundational domain knowledge of the core engine behind all of modern AI.

Project Scope: The "From Scratch" Mandate

The Mandate

The lab provides a clear, challenging scope. We are explicitly prohibited from using high-level PyTorch modules:

  • torch.nn (e.g., no `nn.Linear`, `nn.LayerNorm`)
  • torch.nn.functional
  • torch.optim (e.g., no `optim.AdamW`)

This forces implementation of every component.

What We Will Build

This project requires us to code every single piece of the pipeline:

  • BPE Tokenizer (The BPE algorithm)
  • Transformer Model (RMSNorm, Causal Attention, RoPE, SwiGLU FFN)
  • Training Mechanics (Cross-Entropy Loss, AdamW Optimizer, LR Scheduler)
  • The full training loop, generation, and ablation studies.

Data Availability

TinyStories Dataset

  • Size & Structure: 2.1 million simple children's stories. Provided as a single, large plaintext file.
  • Cleanness: Very clean, synthetic data. Excellent for rapid development, debugging, and initial training.
  • Use: Train the BPE tokenizer and our first model. Can be run on a laptop (CPU/MPS).

OpenWebText Dataset

  • Size & Structure: Large-scale web text corpus. Also a single plaintext file.
  • Cleanness: Unfiltered web data. It will be noisy and contain real-world biases, toxicity, and misinformation.
  • Use: Full-scale model training and evaluation against a leaderboard.

Feasibility & Limitations

Feasibility

This project is highly feasible for three main reasons:

  • Guided Structure: The handout acts as a perfect roadmap, breaking the project into manageable sections.
  • Low-Resource Tips: The lab provides specific advice for running on personal hardware (like an M1 Mac or CPU) using the TinyStories dataset.
  • Clear Scope: The "from scratch" mandate, while challenging, clearly defines the boundaries of the project.

Limitations & Concerns

The primary challenges and concerns are:

  • Implementation Complexity: Building from scratch is error-prone. A single math bug can cause silent failures.
  • Compute Limits: The full lab is designed for H100 GPUs. Scaling on personal hardware will be a bottleneck.
  • Data Bias: The OpenWebText model will inherit all the biases and toxicity of the unfiltered web.
